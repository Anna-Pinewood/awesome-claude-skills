# Obsidian Review — дизайн минимум-версии

Дата: 2026-07-21
Статус: утверждён (брейнсторм-сессия с Claude Code)

## Цель

Когда агент Claude Code правит файлы Obsidian-волта, пользователь автоматически
получает ревью прямо в Obsidian: изменённые страницы открываются сами, удалённое
подсвечено красным, добавленное — зелёным, и правки можно принять или отклонить
на трёх уровнях — отдельное изменение, файл, всё разом. После принятия подсветка
и память о старом тексте исчезают.

Ключевое требование: система работает **детерминированно, через хуки Claude
Code**, а не через «модель прочитала скилл и не забыла». Агент может вообще не
знать о существовании ревью — его правки всё равно будут пойманы и показаны.

## Контекст окружения

- Волт: `/Users/olgalipina/Yandex.Disk.localized/obsidian-vault/cloud-base`
  (~470 md-файлов, ~4 МБ текста; синкается Яндекс.Диском; пользователь волтом
  пользуется одна).
- Волт — НЕ git-репозиторий, и git внутрь синк-папки класть нельзя
  (риск порчи `.git` синк-сервисом — известная проблема).
- Агент правит волт разными инструментами: `mcp__obsidian__edit`,
  `mcp__obsidian__vault`, а также обычными Edit/Write по абсолютному пути.
- В Obsidian уже стоит плагин semantic-vault-mcp с localhost-HTTP на 3001 —
  повторяем его паттерн (loopback + bearer-токен).

## Архитектура

Три компонента:

1. **Хуки** в `~/.claude/settings.json` (user-level — работают из любой сессии):
   - `PreToolUse` — снимает базлайн перед первой правкой задачи;
   - `Stop` — в конце хода агента считает дифф и отправляет плагину.
2. **Скрипты + shadow-git** (`scripts/`): владеют git-архивом
   `~/.claude-obsidian-review.git`, который живёт **вне волта** (Яндекс.Диск его
   не видит) и работает с волтом через `--git-dir/--work-tree`.
3. **Obsidian-плагин** (`plugin/`): принимает пакет диффов по HTTP, открывает
   страницы, рендерит подсветку и кнопки accept/reject.

### Поток данных

```
пользователь: «сходи в обсидиан, допиши X»
  → агент вызывает пишущий инструмент
      → PreToolUse: [маркера нет] git-коммит «фото до» + создать маркер
  → агент делает N правок (PreToolUse видит маркер и молчит)
  → агент закончил ход
      → Stop: [маркер есть] собрать изменённые .md
              → достать старые версии из базлайн-коммита
              → POST пакет плагину (localhost)
              → при успехе: коммит «фото после», удалить маркер
              → при неудаче (Obsidian закрыт): ничего не коммитить,
                маркер оставить — следующий Stop дошлёт накопившееся
  → плагин: Notice «Claude изменил N файлов», открывает вкладки,
    подсветка + кнопки
  → пользователь принимает/отклоняет; принятое исчезает из памяти плагина
```

## Ключевые решения и почему

| Решение | Альтернативы, которые отвергли | Почему |
|---|---|---|
| Базлайн = shadow-git вне волта | Полная копия волта в памяти плагина; хэш-индекс; CriticMarkup-разметка в файлах; git прямо в волте | Хэш не хранит старый текст; копия в RAM не масштабируется и живёт только до рестарта; CriticMarkup виден dataview/publish и требует безошибочной разметки от LLM; `.git` в синк-папке Яндекс.Диска рискует поломкой репозитория. Shadow-git: сжатое хранение на диске, дедупликация по хэшу, дельты, история всех сессий бонусом, ноль служебных файлов в волте. |
| Оркестрация хуками, не скиллом | Скилл-дирижёр, который просит модель дёргать begin/push | Хуки выполняются харнессом детерминированно; «агент обязательно открывает изменённые страницы» становится гарантией системы, а не надеждой на дисциплину модели. |
| Дифф считает `@codemirror/merge` | Свой формат ханков, парсинг `git diff` | Пакет плагину — просто «путь + старое содержимое целиком». Библиотека сама считает дифф, рендерит подсветку и даёт готовые accept/reject по ханкам. Не изобретаем ни формат, ни логику. |
| Состояние ревью — в памяти плагина, только текущая сессия Obsidian | Персистентность на диске, синк на другие устройства | Выбор пользователя. Закрыть Obsidian с непринятым диффом = «принять всё по умолчанию» (правки уже в файлах). История остаётся в git-архиве. |

## Компонент 1: хуки и git-механика

### Маркер «задача идёт»

Пустой файл `~/.claude-obsidian-review.git/task-active`. Нужен, потому что
PreToolUse срабатывает перед **каждой** записью, а «фото до» надо снять ровно
один раз — перед первой. Жизненный цикл: создаётся при первой правке задачи,
удаляется при успешной отправке ревью в Stop. При неудачной отправке живёт
дальше — правки копятся против старой точки до первого успешного push.

### PreToolUse (`ensure-baseline`)

- Матчер: `mcp__obsidian__edit`, `mcp__obsidian__vault`, `Edit`, `Write`.
- Скрипт читает JSON вызова со stdin и выходит мгновенно, если: путь вне волта,
  действие не пишущее (для `vault` — read/list/search и т.п.), или маркер уже
  стоит.
- Иначе: `git add -A && git commit` (текущее состояние волта, включая все
  ручные правки пользователя с прошлой задачи, становится точкой отсчёта),
  создать маркер.
- Хук блокирующий, но быстрый: проверка грязности дерева на ~500 файлах —
  десятки мс; полный первый коммит — ~1 с однократно.

### Stop (`push-review`)

- Маркера нет → мгновенный выход (обычные разговоры не платят ничего).
- Маркер есть:
  1. `git add -A`; список изменённых `.md` относительно HEAD;
  2. для каждого — старое содержимое из `git show HEAD:путь`
     (новый файл → пустая строка; удалённый — помечается отдельно);
  3. POST JSON `{files: [{path, baseline, deleted?}]}` на
     `http://127.0.0.1:<port>/review` с bearer-токеном;
  4. успех → `git commit` («фото после»), удалить маркер;
     неудача → не коммитить, маркер оставить, вывести сообщение
     «Obsidian закрыт, ревью не показано; правки применены».

### Отклонение правок и git

Reject в плагине пишет старый текст обратно в файл на диске. Для git это
неотличимо от ручной правки пользователя — уедет в базлайн при следующей
задаче. Обратной связи плагин→git нет и не нужно.

### Что трекается

Только `*.md`. `.obsidian/` и вложения исключены (через `info/exclude`
shadow-репозитория). Путь волта — константа в конфиг-файле скриптов.
Разовая инициализация — `setup.sh` (создаёт shadow-repo, exclude, первый
коммит, печатает блок хуков для settings.json).

## Компонент 2: Obsidian-плагин

### Приём

HTTP-сервер на loopback, порт в настройках (по умолчанию 3002), bearer-токен
(генерируется при первом запуске, показывается в настройках — его же прописать
в конфиг скриптов). Один endpoint `POST /review`.

### Состояние

`Map<путь, оригинал>` в памяти + производные диффы. Ничего не пишется ни в
волт, ни на диск. Рестарт Obsidian очищает всё.

Если приходит пакет по файлу, который **уже в ревью** (агент сделал две задачи
подряд без принятия), плагин **сохраняет старый оригинал** — подсветка
накапливается, пользователь видит суммарный дифф со своей последней принятой
точки. Молчаливых потерь нет.

### UI

- Notice «Claude изменил N файлов» при приёме пакета.
- Каждый файл открывается в отдельной вкладке в Live Preview.
- **Inline-режим** (основной): `unifiedMergeView` из `@codemirror/merge` как
  editor extension — удалённое красным (блок старого текста), добавленное
  зелёным, у каждого ханка (группы подряд идущих изменённых строк) кнопки ✓/✗.
- **Side-by-side**: переключатель в шапке вкладки; `MergeView` — слева базлайн,
  справа текущий текст, markdown как исходник (как diff в VS Code), те же ✓/✗
  по ханкам.
- **Три уровня, на каждом обе операции:**
  | Уровень | Принять | Отклонить | Где |
  |---|---|---|---|
  | Ханк | ✓ у ханка | ✗ у ханка | inline и side-by-side |
  | Файл | «Принять файл» | «Отклонить файл» | шапка вкладки |
  | Всё | «Принять все правки» | «Отклонить все правки» | палитра команд; клик по статус-бару «Ревью: N файлов» |
- Семантика: принять = подсветка исчезает, старый текст забыт; отклонить =
  старый текст записывается обратно в файл. Принять/отклонить файл = то же для
  всех ханков файла, файл выходит из ревью. Когда ревью пусто — статус-бар
  очищается.

### Edge-кейсы

- **Пользователь правит файл во время ревью**: дифф живой (против оригинала),
  ручные правки тоже подсветятся. Осознанно не боремся — пользователь одна.
- **Созданный файл**: оригинал пуст, весь текст зелёный; reject удаляет файл.
- **Удалённый файл**: страницу не открыть — строчка в Notice («удалён X»),
  восстановление только руками из git-архива.
- **Только frontmatter изменён**: в Live Preview свойства рендерятся виджетом,
  подсветка может не отобразиться — смотреть в side-by-side/source. Известное
  ограничение.

## Не входит в минимум

- Комментарии и вопросы к строкам; любая отправка данных обратно в Claude Code
  (это максимум-версия; localhost-канал плагина — задел под неё).
- Подсветка в Reading mode и в виджете свойств.
- Персистентность ревью между перезапусками Obsidian; другие устройства; mobile.
- Не-markdown файлы и вложения.
- Восстановление удалённых файлов кнопкой.
- Параллельные ревью от двух одновременных сессий Claude Code (один общий
  маркер; кейс экзотический).

## Тестирование (минимальное)

- **Скрипты**: один smoke-скрипт на одноразовом волте в temp:
  «первая правка снимает фото», «десять правок не двигают фото»,
  «push не удался → флаг жив, коммита нет», «успешный push → новый базлайн,
  флага нет». Запуск руками при изменении скриптов.
- **Плагин**: ручной сквозной чек-лист — агент правит 2–3 файла (включая один
  новый) → вкладки открылись, подсветка верна → принять ханк, отклонить ханк,
  принять файл, отклонить всё → файлы и подсветка ведут себя как заявлено.
  Юнит-тестов на UI нет.

## Критерий готовности

Сквозной сценарий «попросила дописать → страницы сами открылись с
красным/зелёным → приняла часть, отклонила часть, приняла всё» работает без
единого ручного действия между просьбой и ревью.
