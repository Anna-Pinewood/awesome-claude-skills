# decomposition-deck — визуальная декомпозиция

Цель скилла — собрать **один html-файл** по готовой декомпозиции брифа, чтобы команда за 15–20 минут увидела целиком: что вообще мы хотим сделать и зачем, какие блоки, сколько sp, где RnD, где блокеры, какие цитаты из брифа стоят за оценками. Это последний шаг в воронке оценки: погружение → вопросы → декомпозиция → визуализация.

Важная установка: **не все, кто будет смотреть деку, погружены в бриф**. Техлиды и смежные команды могут прийти на встречу с нулевым контекстом. Поэтому первое, что должна давать дека, — короткое введение «какая боль, что добавляем, образ результата», потом блоки с оценками.


## Главный принцип №1: оценки — с привязкой

Артефакт делается ради того, чтобы команда поняла **моё виденье решение проблемы в виде задач** и **почему здесь именно столько sp**. Каждая нетривиальная оценка должна быть привязана к одному из трёх источников:

- **цитата из брифа** (если в decomposition.md она уже есть — копируем; если есть бриф/Q&A на входе — ищем подходящую и подкрепляем);
- **RnD-пункт** (исследование, которое и делает блок дорогим);
- **блокер / зависимость** (внешний фактор, удлиняющий оценку).

Если оценка без привязки — это валидный случай (решение принято на интуиции/опыте), но визуально такие задачи **никак не подсвечиваем** — просто строка с sp. Не выдумывай привязку, если её нет.

## Главный принцип №2: ничего не выдумывать

Скилл **пересказывает decomposition.md в визуальной форме**, не редактируя содержание:

- Оценки в sp — дословно из markdown.
- Формулировки блоков/подзадач — дословно или с минимальным сжатием для длины строки. Не переписывай.
- Блокеры, RnD, открытые вопросы — копируются как есть.
- Цитаты из брифа/Q&A — дословные, со ссылкой на источник.

**Исключение — intro-секция** (боль/решение/фичи/образ результата). Её содержание собирается на основе предыдущих артефактов – брифа, дека, созвонов. Подводка может быть твоей формулировкой, но смысл — только из брифа.

## Вход и выход

**Вход:**
1. `decomposition.md` — обязательный, центральный артефакт.
2. Бриф — для подбора цитат-объяснений к блокам.
3. Q&A / треды / записи звонков (опционально) — дополнительные источники цитат.

**Выход:** один html-файл `<decomposition-basename>.deck.html` рядом с decomposition.md (или по явно указанному пути). Файл открывается двойным кликом, работает офлайн.

## Workflow

### Шаг 1. Прочитать decomposition.md

### Шаг 2. Собрать материал для intro

Intro нужен **всегда**. Цель — за 1–2 экрана дать читателю, который не открывал бриф, ответы на три вопроса:

1. **Какая боль** — что не так сейчас, у кого болит, что без этого происходит.
2. **Решение и фичи** — что мы добавляем/меняем, чтобы боль закрыть.
3. **Образ результата** — как это будет выглядеть/работать с точки зрения пользователя или системы, когда сделаем. Акцент на то, что мы добавляем, если продукт уже работает, а не на том, что уже есть.

Откуда брать материал:

- **Бриф** — иди по разделам «Проблема/Мотивация/Цель» (→ боль), «Решение/Подход/Гипотеза» (→ решение), «Фокус/Scope/Что делаем» (→ фичи), сценарии и мокапы (→ образ результата). Извлекай дословные цитаты.
- **Промежуточные артефакты** – здесь лежат уже осмысленные и дополненные артефакты, особенно записи звонков и треды, там будут прямые цитаты продакта.

Важно смотреть артефакты в сумме, сам по себе бриф не является источником истины. Более поздние артефакты имеют больший вес.

### Шаг 3. Спроектировать структуру

Структура — фиксированная рамка, внутри которой блоки идут **жёстко 1:1 с decomposition.md**:

1. **Cover** — название, дата, суммарный sp, число блоков.
2. **Intro** — «Какая боль», «Что добавляем / решение», «Образ результата». 1–2 секции, ~1–2 экрана. Единственная часть деки, которой нет в decomposition.md.
3. **Sp-сводка** — все блоки в одной visual-сводке (табличка).
4. **Блоки** — по одной секции на блок, **в порядке из decomposition.md**.

Кроме intro — никаких других «придуманных» секций («риски в целом», «timeline», «план релизов»). Этого нет в decomposition.md, значит нет и в html.

### Шаг 4. Сгенерировать html

**Перед генерацией загляни в референсы стиля** в `/Users/olgalipina/Documents/awesome-claude-skills/html-style/`:

- `09-slide-deck.html` — плотность секций, типографика, способ подачи ключевых тезисов.
- `14-research-feature-explainer.html` — длинный скролл-документ с секциями, цитатами, карточками; самый близкий по жанру.
- `13-flowchart-diagram.html` — стиль аннотированных схем, **только если** в decomposition.md есть явные зависимости между блоками и ты решил их нарисовать.

Из референсов **копируй**: палитру, типографику, отступы, плотность, стиль карточек / цитат / разделителей секций / sticky-навигации. **Не копируй**: содержание, тематику.

Требования:

- **Один файл.** Всё inline: CSS в `<style>`. Открывается двойным кликом, работает офлайн.
- **Скролл, не слайды.** Вертикальный поток секций. Граница между блоками визуально чёткая.
- **Sticky-навигация сверху или сбоку** — короткий список блоков (по заголовкам из decomposition.md), клик скроллит к блоку. В длинной декомпозиции без неё теряешься.
- **Sp визуально подчёркнут.** Цифра sp у подзадачи — крупнее/жирнее обычного текста. «Итого» по блоку — ещё крупнее. Суммарный sp на cover — самый крупный.
- **Цитаты из брифа — визуально отличимы** (левая полоска, моноширинный или сериф, приглушённый фон). Каждая цитата сопровождается ссылкой: `бриф, §<секция>` / `Q&A, вопрос N`.
- **Якорные слова в длинных цитатах.** Если цитата длиннее ~15 слов — выдели 2–4 ключевых слова (`<mark>` или собственный класс с приглушённым акцентным фоном). Не больше 4 хайлайтов на цитату.
- **Никаких внешних зависимостей.** Если нужны бары/стрелки — рисуй inline SVG или используй CSS. Не подключай chart.js / mermaid / прочее.


### Шаг 5. Записать артефакт и открыть html

По умолчанию: рядом с decomposition.md, имя `<decomposition-basename>.deck.html`.

Пример: `claude-tasks/q3-design/decomposition_buy-agent.md` → `claude-tasks/q3-design/decomposition_buy-agent.deck.html`.

Если пользователь указал другую директорию или путь — клади туда.

В финальном сообщении пользователю:
- путь к созданному файлу;
- если на вход не передавали бриф/Q&A — упомянуть, что цитат-объяснений в html нет (можно перегенерировать с брифом).
- если для каких-то блоков не нашлось цитат — это нормально, не оправдывайся.
