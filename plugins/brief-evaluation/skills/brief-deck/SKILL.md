# brief-deck — визуальный пересказ брифа

Цель скилла — собрать **один html-файл**, который инженер открывает вместо брифа и за 3–5 минут получает картину: какую проблему решаем, как продакт предлагает её решить, что уже есть, где в брифе нестыковки.

Скилл — подспорье человеку.

## Главный принцип №1: плотность важнее полноты

Бриф — длинный. Артефакт **brief-deck — более ёмкий**. Цель не пересказать бриф, а **подсветить суть**, чтобы инженер за 5-10 минут собрал в голове картину. Важно использовать цитаты, но "умно", урезать повторы одной и той же мысли, что-то группировать.

**Исключение — секция «Противоречия и пробелы».** Её НЕ ужимай: каждое противоречие даётся в полном виде (две цитаты + строка «в чём конфликт»), даже если их много.

**Правило больших чисел:** если для секции набралось >8 пунктов — это плохо извлечённая суть, а не богатый бриф. Сгруппируй или укрупни.

## Главный принцип №2: ничего не выдумывать

Скилл **пересказывает то, что есть в брифе**, и подсвечивает противоречия. Он НЕ:
- додумывает мотивы продакта;
- фиксирует «своё мнение» о решении;
- предлагает альтернативы;
- даёт инженерные оценки.

Если в брифе чего-то из обязательных секций нет (например, не сформулирована проблема) — это и есть честный результат: секция «Проблема» содержит пометку `// в брифе не сформулировано явно` и 1–2 цитаты, из которых можно догадываться.

Обобщать **можно**, если одна мысль размазана по брифу в 5–10 местах. В этом случае извлеки её одним абзацем и пометь «собрано из разных мест брифа». Желательно — давать прямые цитаты блоков из брифа.

## Вход и выход
Вход: **Бриф** в любом формате, опциональный доп. контекст
Выход: создан файл, скил открывает `.html`. Положить рядом с брифом, имя `<brief-basename>.deck.html`.


## Workflow

### Шаг 1. Прочитать бриф и доп. материалы целиком

Один проход, без выписывания. Цель — поймать общую логику и понять, где какие куски лежат. На этом шаге ничего не пишем.

### Шаг 2. Извлечь обязательные блоки

Пройди по брифу второй раз и собери материал под пять обязательных секций. Для каждой секции собирай **цитаты + ссылки на место в брифе**, а не свои пересказы.

1. **Проблема.** Что болит у пользователя/бизнеса. Ищи в секциях про мотивацию, цель, гипотезу, «зачем». Если проблема явно не сформулирована — извлеки косвенные сигналы и помечай как реконструкцию.
2. **Как хотим решить.** Подход, гипотезы, направления. Что именно предлагает продакт. Не реализация, а замысел.
3. **Фичи / скоуп.** Список конкретных пунктов функционала. Иногда есть отдельная секция («Фокус», «Что делаем», «Scope»), иногда — собирается из сценариев, мокапов, описаний флоу. Отдели **новое** от **уже существующего**, если бриф — продолжение проекта.
4. **Что уже есть.** Только если есть доп. контекст (код / прошлые звонки) или же бриф ссылается на предыдущие версии. Если ничего такого не упомянуто – опусти секцию.
5. **Противоречия и пробелы.** Места, где бриф сам себе противоречит, либо где блок упомянут в одной части и упущен в другой (например, есть в мокапе, нет в «Фокусе»; есть в метриках, не описан в сценариях). Каждое противоречие — с двумя цитатами и одной строкой «в чём конфликт».

### Шаг 3. Решить, какие визуализации нужны

Скилл — это **не текстовый пересказ в html**. Каждая секция должна иметь визуальный якорь, но **только если он реально помогает понять**. Не вставляй декоративные карточки ради эстетики.

Допустимые визуализации:

- **Список-карточки.** Для фич/скоупа — один пункт = одна карточка с заголовком, цитатой из брифа, одной строкой пересказа.
- **Two-column diff.** Для «уже есть vs хотим добавить», для «было в мокапе / нет в фокусе».
- **Flowchart.** **Только** если в брифе явно описан поток данных или пользовательский сценарий с шагами (1 → 2 → 3), и у каждого шага есть свои подробности. Если есть описание «пользователь вводит запрос → агент уточняет → возвращает выдачу → клик ведёт на карточку» — рисуй. Подписывай узлы цитатами/короткими формулировками из брифа. Помечай реконструированные шаги отдельно.
- **Подсветка-цитата.** Большая цитата из брифа, когда формулировка сама по себе важна.
- **Conflict-блок.** Два контрастных бокса рядом + одна строка «в чём конфликт».

### Шаг 4. Спроектировать структуру (не больше 10 секций)

Базовый каркас (порядок фиксированный):

1. **Cover** — название брифа, дата, 1 строка-«твит» о чём это.
2. **Проблема.**
3. **Как решаем** (подход / гипотезы).
4. **Что уже есть** — если применимо, иначе пропустить.
5. **Фичи / скоуп** — может занять 2–3 секции, если пунктов много (разбей по группам, не по одному пункту на секцию).
6. **Ключевой сценарий / flowchart** — если есть.
7. **Противоречия и пробелы.**
8. **Источники** — список путей к брифу и доп. материалам, дата сборки, версия.

Всего ≤ 10 секций. Если получается больше — укрупняй, не дроби.

### Шаг 5. Сгенерировать html

**Перед генерацией загляни в референсы стиля** в `/Users/olgalipina/Documents/awesome-claude-skills/html-style/`:

- `09-slide-deck.html` — слайдоподобная плотность секций, типографика, способ подачи ключевых тезисов.
- `13-flowchart-diagram.html` — стиль аннотированных схем (узлы, стрелки, подписи). Используй как образец, если у тебя в брифе действительно есть пошаговый сценарий.
- `14-research-feature-explainer.html` — длинный скролл-документ с секциями, цитатами, карточками; самый близкий по жанру к brief-deck.

Из референсов **копируй**: палитру, типографику, отступы, плотность, стиль карточек / цитат / разделителей секций / sticky-нужции / стрелок в SVG. **Не копируй**: содержание, конкретные тексты, тематику.

Если визуальное решение для какого-то блока неочевидно — выбери ближайший аналог из референсов и сделай по нему. Не изобретай новый стиль.

Требования:

- **Один файл.** Всё inline: CSS в `<style>`. Файл должен открываться двойным кликом и работать офлайн.
- **Скролл, не слайды.** Вертикальный поток секций. Каждая секция занимает примерно высоту экрана или меньше, чёткая визуальная граница между секциями (фон / разделитель).
- **Sticky-нужция сверху или сбоку** — короткий список секций, клик скроллит к ней. Без неё в длинном документе теряешься. **Активная секция должна быть визуально подсвечена**, чтобы по навигации сразу было понятно, на каком блоке сейчас скролл. Подсветка обновляется при скролле — через `IntersectionObserver` на секциях, без внешних библиотек.
- **Простота.** Светлый фон, один акцентный цвет, читаемая типографика (system-ui / -apple-system), max-width контента около 760–900px, ширина flowchart-секций может быть больше.
- **Цитаты из брифа — визуально отличимы** (левая полоска, моноширинный или сериф, приглушённый фон). Каждая цитата сопровождается ссылкой на источник: «бриф, §<название секции>».
- **Якорные слова в длинных цитатах.** Если цитата длиннее ~15 слов — выдели в ней 2–4 ключевых слова/коротких словосочетания (`<mark>` или собственный класс с приглушённым акцентным фоном), чтобы взгляд цеплялся за смысловые опоры и можно было прочитать «по диагонали». Выделяй существительные/действия, несущие смысл (что делаем, на кого, какой эффект), а не связки. Не больше 4 хайлайтов на цитату — иначе перестаёт работать.
- **Никаких внешних зависимостей.** Если нужен flowchart — рисуй inline SVG (узлы-прямоугольники, стрелки-линии). Не подключай mermaid/d3/прочее.
- **Пометки реконструкции.** Если блок собран из разных мест или додуман — визуально пометить (тонкий лейбл «реконструкция» / «собрано из разных мест»).


### Шаг 6. Записать артефакт

По умолчанию: рядом с брифом, имя `<brief-basename>.deck.html`.

Если пользователь указал другую директорию или путь — клади туда.

В финальном сообщении пользователю:
- путь к созданному файлу;
- что **не получилось** или какие блоки пришлось пометить как реконструкцию.
