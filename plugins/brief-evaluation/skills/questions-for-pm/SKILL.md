# questions-for-pm — план разговора с продактом по брифу

Цель скилла — собрать **два markdown-файла**: полную версию для созвона с продактом и короткую выжимку на лист A4 для случая, когда у продакта мало времени. Оба готовы к использованию без редактуры; после созвона полная версия служит входом в следующую стадию оценки.

Скилл – подспорье человеку.

## Что нужно для старта

1. **Бриф** — абсолютный путь к файлу с текстом брифа.
2. **Контекст команды** — путь к `team.md` (статика: компетенции, что берём / не берём, ожидания от других команд).
3. **Контекст проекта**, например:
   - путь к корню кодовой базы существующего проекта, к которому бриф относится;
   - путь к артефактов в `claude-tasks/<relevant-task>/...` со сводками, которые могут относится к задаче
   - записи прошлых звонков
   - любой исторический контекст

**Если чего-то не хватает — задай ОДИН атомарный вопрос и остановись.** Не начинай работу с неполным входом — это первый источник мусорных вопросов.

## Главная идея: смотреть на бриф как на сценарий

Бриф описывает фичу или техническое изменение, но и то и другое существует, чтобы появиться в каком-то **сценарии** — последовательности шагов и развилок, через которую проходит запрос / событие / действие. Поэтому ключевая часть скилла — не пройтись по списку «функциональных компонентов», а **восстановить этот сценарий** и идти по нему. Так находятся пропущенные шаги, необработанные ветки, неучтённые состояния — то, что в разбиении «по фичам» обычно теряется.

«Сценарий» — в самом широком смысле. Это **не обязательно** путь конечного пользователя в приложении. В зависимости от типа брифа «актор» сценария меняется:

- **Конечный пользователь приложения** — самый частый случай: вошёл → увидел саджесты → ввёл текст → подтвердил → отправил.
- **Внутренний пользователь** (дежурный инженер, оператор, аналитик) — для брифов про мониторинг, админки, внутренние инструменты. Например: дежурный получил алерт → открыл дашборд → увидел метрику → проверил логи → принял решение.
- **Запрос / событие в пайплайне** — для брифов про инфраструктуру, миграции, переход компонента с одной модели/сервиса на другой. Например, перевод компонента с Otello на Гигачат: запрос пришёл в шлюз → новый адаптер → новая модель → постпроцессинг → ответ → старый путь выключился.
- **Тренировочный / деплойный цикл** — для брифов про ML / инфру: данные собрались → отфильтровались → попали в тренировку → метрики посчитались → версия выкатилась → откат при деградации.

В любом из этих случаев структура одна и та же:

- **Шаги** — последовательные стадии пути актора.
- **Ветки** — развилки внутри шага: что если актор сделал X / Y / Z; что если данные содержат A / B; что если предыдущий шаг вернул успех / ошибку / пусто.
- **Состояния** — что происходит при граничных событиях: актор прервал поток и вернулся позже; данные не соответствуют формату; смежная система недоступна; контекст пустой или противоречив.

«Лимиты фото», «состояние между сессиями», «авторизованные vs неавторизованные», «что если Гигачат недоступен» — это **не мета-вопросы**, а ветки/состояния сценария, привязанные к конкретным точкам.

Истинно мета-вопросы (не про сценарий) — редкость. Это: статус брифа (зачищен ли после разделения), capacity и контракт с другими командами, кто пишет тексты, владельцы внешних систем. Продуктовые вопросы (KPI, метрики, пилот) — отдельный класс, идут в самом низу артефакта.

## Workflow


### Шаг 1. Прочитать вход

Прочитай сопроводительные документы. Не уходи в глубокое исследование кода — нужен только базовый каркас, чтобы понимать терминологию брифа.

**Важно**: далее по воронке вопросов у тебя могут появится вопросы по сценариям. Если ты предполагаешь, что снимки кодовой базы / документации могут тебе помочь – исследуй с прицелом на конкретное уточнение, не ищи просто так, но и не упускай этот источник информации.

Зафиксируй внутренне:
- цель и проблема, заявленные в брифе;
- **сценарий** — если в брифе есть явный (раздел «Сценарий», иллюстрации, таблица флоу, схема пайплайна) — выпиши его шаги. Если явного нет — реконструируй из списка фич или технических изменений: каждое существует, чтобы появиться в каком-то моменте сценария, по ним можно собрать путь. Определи актора: конечный пользователь, внутренний пользователь, запрос в пайплайне, тренировочный цикл. В сценарной секции потом явно скажешь продакту: «вот наша реконструкция шагов — подтверди или поправь»;
- упоминания других команд, внешних систем, дат, цифр;
- что из брифа ты не поняла механически (термины, которые не маппятся на код / прошлый опыт).

### Шаг 2. Высокоуровневый разбор (problem & goal)

Найди в брифе разделы про цель, проблему пользователя, гипотезу, влияние. **KPI и метрики сюда НЕ включаются — они идут в шаг 3c (продуктовые вопросы) в конце артефакта.**

Эта секция почти всегда состоит только из **подтверждений вижна** — тезисы вытаскиваешь из брифа, формулируешь одним предложением и просишь продакта подтвердить, что вы on the same page.

Формат:
> «Мы понимаем цель этапа как X. Подтверди.»

Что сюда входит:
- цель MVP / этапа в одной фразе;
- главная пользовательская боль, которую решаем;
- границы скоупа MVP (что делаем / не делаем);
- граница ответственности нашей команды vs других.

Отдельной секции «Уточнить» здесь нет. Если по высокоуровневому тезису возник вопрос — он почти всегда либо про конкретный шаг сценария (→ шаг 3), либо про процесс/команды (→ шаг 3b), либо про KPI (→ шаг 3c).

**Запрещено** в этой секции:
- Вопросы про существование всей фичи целиком («а нужен ли вообще товарный агент»). Не наша зона ответственности — это работа продактов.
- Шаблонные риск-вопросы («какие риски?», «как замерим?»).
- Технические вопросы про реализацию («какой алгоритм ранжирования»).

### Шаг 3. Сценарный разбор

Это основная часть артефакта. Пройди по сценарию (восстановленному в шаге 1) **шаг за шагом и ветка за веткой**. Для каждой осмысленной точки сценария построй карточку.

**Что такое «осмысленная точка»:**
- Шаг сценария (для пользователя в приложении: вход / увидел саджесты / ввёл текст / подтвердил / отправил; для пайплайна: запрос пришёл / прошёл препроцессинг / попал в модель / постпроцессинг / ответ; для дежурного: алерт / дашборд / логи / решение).
- Ветка внутри шага («Другое», «Пропустить», переключение на запасной путь, smart fallback, нет результата).
- Состояние / событие актора (пользователь бросил диалог и вернулся; запрос пришёл с битыми данными; модель вернула timeout; смежная система недоступна; контекст пустой).
- Граничный случай данных (контекст противоречив; в карточке нет нужных полей; в данных пустые ключи; формат не соответствует контракту).

**Если в брифе есть только список фич / техизменений, без сценария** — реконструируй сценарий из них и явно об этом скажи в начале секции:

> _Сценарий ниже — наша реконструкция по списку <фич / технических изменений> в брифе. Актор — <конечный пользователь / дежурный / запрос в пайплайне>. Подтверди, что поток выглядит примерно так, или поправь._

**Правило отбора:** если в осмысленной точке сценария **нет утверждения в брифе** — это карточка с вопросом. Если есть, но размытое или противоречит другому — тоже карточка. Если есть и ясное — короткая карточка с подтверждением интерпретации.

Не разбирай функционал, который уже в продакшене и не меняется.

**Карточка шага/ветки:**

```
### <короткое имя шага/ветки>

**В брифе:**
> <дословная цитата ИЛИ "явного описания нет — реконструируем из <ссылка на пункт>">

**Что происходит в этой точке:**
<1-3 предложения: что видит / делает актор сценария (пользователь / дежурный / запрос в пайплайне) в этой точке>

**Наша интерпретация:**
<1-3 предложения: как мы поняли механический смысл, опираясь на код / историю проекта>

**Вопросы (если есть):**
- <вопрос 1 с контекстом>
- <вопрос 2 с контекстом>
- ...
```

**Правило формулировки вопроса:** каждый вопрос должен соответствовать **хотя бы одному** из шести критериев полезности (ниже). Если ни один не подходит — выкинь вопрос.

#### Критерии полезного вопроса (применяй в шаге 3)

1. **Способен убить или урезать пункт скоупа.** Постановка проверяет посылку, и если посылка ложная — пункт отваливается.
2. **Вскрывает альтернативу из истории проекта, которой нет в брифе.** Раньше обсуждалось другое решение / другая интеграция — почему сейчас иначе.
3. **Требует конкретики там, где формулировка размыта.** Просишь примеры, определения, границы, состав сущности.
4. **Проверяет совпадение нашей инженерной интерпретации с продуктовой.** Озвучиваешь свою интерпретацию явно и просишь подтвердить.
5. **Выясняет мотивы, не указанные в брифе.** За пунктом может стоять договорённость с другой командой, требование сверху, прошлый эксперимент — без этого мы либо начнём резать политически важное, либо примем неверную интерпретацию.
6. **Закрывает дыру в шаге или ветке сценария.** В этой точке пути актора бриф молчит, а решение там обязательно должно быть. Что происходит? Что если предыдущий шаг вернул X / Y / пусто? Что после успеха, что после ошибки, что при недоступности смежной системы? Особенно — на ветвлениях, граничных событиях и состояниях.

**Каждый вопрос обязательно с контекстом.** НЕ «почему суточное обновление?». ДА: «В брифе указано суточное обновление атрибутов. Если данные на стороне поставщика меняются реже — мы делаем лишнюю работу; если чаще — отставание накапливается. Откуда взялась эта частота?»

### Шаг 3b. Процессные / межкомандные вопросы

Очень узкая секция, **часто пустая**. Сюда уходит только то, что вообще не привязывается к шагу сценария:

- статус брифа (зачищен ли после разделения, актуальна ли версия);
- зависимости от других команд (capacity UGC на Q2, владельцы внешних систем — YouLa и т.п.);
- кто пишет тексты / делает разметку / отвечает за UI-копирайт (если это размыто в брифе);
- контракт API между нами и смежной командой, если он не привязан к конкретному шагу.

**Если что-то можно привязать к шагу сценария — привязывай и оставляй в шаге 3.** Например, «лимиты файла для фото» — это не процессный вопрос, а ветка «пользователь прикладывает фото».

### Шаг 3c. Продуктовые вопросы (KPI, метрики, пилот)

Отдельная секция в конце разбора. Не главная для этого этапа, но важна для общей картины.

Сюда попадают:
- единица KPI (сессия / пользователь);
- базовые цифры для интерпретации KPI («какая база сейчас», «какая доля попадает в категорию Х»);
- определение «пилота» (процент трафика, длительность, география);
- аналитика и инструментирование воронки;
- что считается «возвратом», «успехом», «выполнено».

Формат — те же карточки с цитатой / интерпретацией / вопросом, что и в шаге 3. Просто отделены тематически.

### Шаг 4. Чистка

Перед записью пройдись по черновику и выкини:
- вопросы, не подходящие ни под один из шести критериев;
- одинаковые шаблоны («какие риски?», «как замерим?») без привязки к месту брифа;
- вопросы про реализацию (алгоритмы, библиотеки, БД-схемы);
- вопросы про существование всего брифа.

Если после чистки в карточке шага/ветки осталось 0 вопросов — оставь карточку с пустой секцией вопросов и пометкой `// нет вопросов — шаг ясен`. Это валидный результат.

### Шаг 5. Записать артефакт

Путь: рядом с брифом, имя `<brief-basename>.questions-for-pm.md`.

Пример: бриф `claude-tasks/q3-desing/docs/brief_may_4.md` → артефакт `claude-tasks/q3-desing/docs/brief_may_4.questions-for-pm.md`.

Шаблон файла:

```markdown
# Вопросы продакту: <название брифа>

Источник: <относительный путь к брифу>
Дата подготовки: <YYYY-MM-DD>

## 1. Подтверди наш вижн

- <тезис 1: цель MVP>
- <тезис 2: главная боль>
- <тезис 3: границы скоупа>
- <тезис 4: граница ответственности>

## 2. Сценарный разбор

_Сценарий ниже — <берём из брифа / наша реконструкция по списку фич / по техизменениям>. Актор — <конечный пользователь / дежурный / запрос в пайплайне / …>. Подтверди, что поток выглядит так, или поправь._

<карточка шага/ветки 1>

<карточка шага/ветки 2>

...

## 3. Процессные / межкомандные вопросы

<карточки. часто пусто — это нормально>

## 4. Продуктовые вопросы (KPI, метрики, пилот)

<карточки по метрикам>

## Заметки после созвона
_<пусто>_
```

### Шаг 6. Короткая версия для занятого продакта

Параллельно с длинной версией собери второй артефакт — выжимку на лист A4 для случая, когда у продакта нет времени на длинный разбор.

Путь: `<brief-basename>.questions-for-pm.short.md` рядом с длинной версией.

**Что входит:**

1. **Подтверждение вижна** (3–5 буллетов): цель этапа, границы скоупа — в той же формулировке, что секция «Подтверди наш вижн» длинной версии, но плотнее.

2. **До 7 вопросов**, отобранных по приоритету (в порядке шагов сценария):
   - **Главное — сценарные блокеры**: вопросы по шагам/веткам, без ответа на которые невозможно понять, что вообще делается с точки зрения пользователя. Вопросы по данным, их источникам, сценарным шагам. Идут в порядке шагов сценария — так продакту проще держать контекст.
   - **Сюда же** — высокоуровневые блокеры (если есть противоречие или дыра на уровне скоупа MVP).
   - **Если осталось место** — 1-2 процессных блокера (например, кто пишет тексты, если без ответа фичу нельзя оценить по срокам).
   - **НЕ включать:** продуктовые/KPI-вопросы. Они идут отдельным разговором (или письмом) — в short.md им не место.
   - **НЕ включать:** уточнения деталей реализации внутри уже понятного шага.

**Формат каждого вопроса:**

```
### <вопрос одной фразой>

<комментарий 1–2 абзаца: почему это блокер, что мы не можем сделать без ответа, какие варианты интерпретации мы видим>

(опц.) > <цитата из брифа>
```

**Жёсткое ограничение:** документ должен помещаться на один лист A4 (~3500 символов основного текста). Если не помещается — режь вопросы снизу (они менее приоритетные по построению).

**Шаблон файла:**

```markdown
# Вопросы продакту (короткая версия): <название брифа>

Источник: <относительный путь к брифу>
Полная версия: <относительный путь к длинному артефакту>
Дата подготовки: <YYYY-MM-DD>

## Подтверди наш вижн

- <тезис 1>
- <тезис 2>
- <тезис 3>

## Главные вопросы по сценарию

<карточка вопроса 1 — по первому шагу сценария>

<карточка вопроса 2 — по следующему шагу>

...
```

## Корпус примеров

Перед формулировкой карточек **прочитай** `${CLAUDE_PLUGIN_ROOT}/skills/questions-for-pm/references/examples.md` — это накопительный архив реальных карточек по живым брифам. Копируй стиль формулировок, плотность вопросов, способ привязки к контексту. Содержание — не копируй.

Если по ходу работы родилась удачная карточка (вскрыла политику, отбросила пункт, выявила противоречие, закрыла дыру в сценарии) — в финальном сообщении пользователю упомяни: «карточку про <имя> стоит добавить в `references/examples.md`». Пользователь сам решит, добавлять ли.

## Анти-примеры (что НЕ писать)

```
- Какова целевая аудитория?
- Какие риски?
- Как мы будем замерять успех?
- Когда дедлайн?
- Точно ли нам нужно ранжирование саджестов?
- Какой алгоритм будем использовать для ранжирования?
```

Каждый из них либо общий шаблон без контекста, либо про реализацию, либо про существование фичи в целом. Все они попадают в чистку (шаг 4).
