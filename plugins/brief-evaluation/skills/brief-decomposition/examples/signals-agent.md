# Декомпозиция: AI-агент для отправки зацепок (signals-agent)

Источники: [бриф](../input_data/brief.md), [Q&A 2026-05-19](../calls/qna_2026-05-19_otpravka-zacepok-cherez-agenta.md), [состав команды](../input_data/team.md), [архитектурные референсы](../input_data/notes_on_agents.md) (slot-filling eval + OfferEngine).

Перспектива — наша команда AI-агентов: логика агента, контракты со смежниками, деплой, наблюдаемость. Внешние зависимости (фронт, UGC, DevOps) фиксируем как точки стыковки, но в скоп нашей команды не включаем.

**Итог: ~29 sp.**

---

### 1. Согласования и контракты со смежниками

- **Когда нужен / зачем:** до старта разработки. Без закрытых контрактов с UGC и фронтом придётся переделывать клиент API и пайплайн контекста. Большая часть пунктов — из Q&A 2026-05-19.
- **Суть работы:** оргработа — звонки с UGC и фронтом, чтение свагера, фиксация договорённостей в Confluence. На выходе — закрытый список изменений в API UGC, схема контракта «контекст вызова» от фронта, доступы в sandbox.
- **Открытые вопросы:**
  - Метка «зацепка от агента» на уровне зацепки в API UGC (нужна, чтобы UGC не запрашивал подтверждающее фото на письма-уточнения) — пометила Алевтина, требуется доработка контракта.
  - Подключение новых источников `Мобильная версия. Агент` / `Онлайн-версия. Агент` на стороне UGC и YouLa.
  - Загрузка фото — через UGC API или отдельный CDN.
  - Формат телефонов в UGC.

Условная разбивка по задачам:
1. Согласования и контракты со смежниками: звонки с UGC (контракт API — метка от агента, новые источники, multi-action, `working_hours`, фото, формат телефонов; sandbox-доступы) + согласование с фронтом контракта «контекст вызова» — **1 sp** *(неделя 1)*
2. Согласования и контракты со смежниками: продолжение — **1 sp** *(неделя 2 — звонки и итерации не помещаются в одну неделю, содержание то же самое)*

Напоминалки (без sp):
- Уточнить у фронта (Илья) тех. возможность кнопки «прикрепить фото» в поле ввода — *напоминалка, не задача*.
- Написать заявку DevOps на сервера и квоты LLM под latency-бюджет 1–2 сек — *напоминалка, не задача*.

**Итого: ~2 sp**

**Зависимости от других команд:** *вся задача — про работу со смежниками: UGC (контракт + sandbox), фронт (контракт контекста + Илья по вопросу фото), DevOps (сервера и квоты).*

### 2. Архитектурная проработка

- **Когда нужен / зачем:** до старта реализации ядра. Без формализованного потока данных и сценариев каждая подзадача в блоке 4 будет одновременно решать «как» и «что» — переделок становится в 2-3 раза больше. Эта задача снимает с реализации всё, что должно быть решено заранее.
- **Суть работы:** формализация потока данных между всеми компонентами (state machine с фазами, slot-schemas по типам зацепок, depends_on и инвалидация, правила маршрутизации саджестов на типы) и формализация пользовательских сценариев для последующей spec-driven разработки (happy-path, multi-intent с порогом «одна `UpdateOrganization` vs. расщепление», fallback-ветки, смена намерения, минимум для зацепки). Согласование сценариев с Алевтиной.
- **RnD часть:**
  - Где проходит порог «одна `UpdateOrganization` с несколькими полями» vs. «расщепление на N зацепок» — правила по типам объектов или эвристика. Решаем в рамках этой задачи, чтобы блок 4 уже шёл по готовым правилам.
  - Адаптация OfferEngine: у нас flat slot sets по типам зацепок и слабые depends_on, не полная state-machine с каскадной инвалидацией. Фиксируем упрощения.
  - Способ маскирования PII (телефоны, имена) перед отправкой в трейсинг — на каком слое; влияет на блок 7.

Условная разбивка по задачам:
1. Формализация потока данных: state-machine, slot-schemas, фазы, правила маршрутизации саджестов — **2 sp**
2. Формализация сценариев под spec-driven разработку (happy-path, ветки, multi-intent, fallback, минимум зацепки) + согласование с Алевтиной — **1 sp** *(может растянуться из-за итераций согласования)*

**Итого: ~3 sp**

**Зависимости от других команд:**
- Алевтина (продакт) — согласование сценариев и развилок.

**Внутренние блокеры:**
- блок 1 в части ответов UGC по типам и `working_hours` (без них slot-schemas повисают).

### 3. Контекст точек входа и саджесты

- **Когда нужен / зачем:** четыре точки входа дают разные саджесты и разный приоритет интентов (из выдачи → `NewOrganization`, с карты → гео, из карточки → атрибуты объекта, бургер → только `NewOrganization`). Out-of-scope-типы (входы, форма зданий) → сразу классическая форма без перехода в чат. Правило приоритета `adm_div` для гибридов.
- **Суть работы:** на вход — контекст вызова от фронта (контракт из блока 1). На выход — нормализованная сущность «точка входа + объект», словарь саджестов и правила маршрутизации саджестов на типы зацепок (правила формализованы в блоке 2). Флаг «уйти в классическую форму без чата» для out-of-scope-саджестов. Тут только код маршрутизации, не решение «что куда».

Условная разбивка по задачам:
1. Парсер контекста от фронта + словарь саджестов «точка входа → 5 + Другое» + маршрутизация на типы зацепок + out-of-scope-флаг — **2 sp**

**Итого: ~2 sp**

**Зависимости от других команд:** нет (контракт с фронтом снимается блоком 1).

**Внутренние блокеры:**
- блок 1, п.2 — финальный контракт контекста от фронта;
- блок 2 — правила маршрутизации саджестов.

### 4. Диалоговое ядро: NLU, slot-filling, state machine

- **Когда нужен / зачем:** основная логика агента — распознать намерение из 4 точек входа, провести slot-filling по типу зацепки, сверить с данными карточки, обработать multi-intent, fallback на классическую форму, парсинг свободной формы (расписание, телефоны, URL), смена намерения по ходу диалога. Тексты ответов агента (приветствия, подтверждения, сверки, благодарности в двух версиях, fallback-сообщения) — часть промптинга в рамках этого блока, отдельной задачей не выделяем.
- **Суть работы:** реализация по схеме из блока 2 (архитектурная проработка). На входе — формализованные state-machine, slot-schemas, правила. На выходе — рабочий агент с собранным payload для UGC-клиента (блок 5).
- **RnD часть:**
  - Выбор модели для intent recognition при SLA 1–2 сек: лёгкая локальная классификация vs. проприетарная LLM. Замеры latency, качества, стоимости.

Условная разбивка по задачам:
1. Скелет state-machine + реализация slot-schemas (по схеме из блока 2) + session store (in-memory для dev, Redis для prod), короткая сессия — **2 sp**
2. NLU: классификатор намерения + extractor сущностей. RnD: выбор модели — **2 sp**
3. Парсеры свободной формы: расписание (7 дней + current values + `0`), телефоны (add/update/delete), URL/соцсети — **2 sp**
4. Сверка с карточкой + подтверждение распознанного намерения + multi-intent + расщепление + fallback-ветки + смена намерения + минимум зацепки + сборка финального payload — **2 sp**

**Итого: ~8 sp**

**Зависимости от других команд:** нет — после блоков 1 и 2 ядро не блокируется снаружи.

**Внутренние блокеры:**
- блок 2 (без формализованных правил и схем оценка вырастет в 2-3 раза);
- блок 1, п.1 (контракт UGC по типам зацепок и multi-action).

### 5. Интеграция с UGC API

- **Когда нужен / зачем:** клиент к `datafeedback-*.api.2gis.ru` с поддержкой всех типов зацепок MVP (`UpdateOrganization`, `NewOrganization`, `NotFound`, `TemporarilyCloseOrganization`, `MapObjectDescriptionError`, `MapObjectError`, `MapPlaceError`, `NoMapObject`, `OtherMapError`, `AZR`), multi-action в одной `UpdateOrganization`, `working_hours` (все 7 дней + `0` для нерабочих), загрузки фото, валидации «текст ИЛИ фото», новых источников и метки «от агента».
- **Суть работы:** http-клиент, сериализаторы по типам, retry-политика, валидация payload, интеграционные тесты против sandbox.
- **RnD часть:**
  - Куда грузится фото — отдельный slot в UGC API или CDN (зависит от ответа UGC по блоку 1).

Условная разбивка по задачам:
1. HTTP-клиент + сериализация по типам зацепок (10 типов MVP) + multi-action `command_user_change` + источник/метка «от агента» — **2 sp**
2. Валидация payload + загрузка фото + ретраи 5xx с бэкоффом + интеграционные тесты против sandbox — **1 sp**

**Итого: ~3 sp**

**Зависимости от других команд:** нет — контракт и sandbox UGC снимаются блоком 1.

**Внутренние блокеры:**
- блок 1, п.1 и п.2 (контракт + sandbox);
- блок 4, п.4 (payload собирает ядро).

### 6. Расчёт нагрузки и подбор железа под локальную модель

- **Когда нужен / зачем:** новых агентов сразу оцениваем с учётом запуска на локальных моделях. Нужно понимать, какую нагрузку модель держит на доступном железе, помещаемся ли в SLA 1–2 сек при ожидаемом потоке (~400k нажатий «Сообщить об ошибке» в месяц по РФ, неравномерно — пики в часы активности), нужно ли заказывать дополнительное железо / GPU.
- **Суть работы:** замеры throughput и latency выбранной модели (см. RnD в блоке 4) на тестовом железе, расчёт пикового RPS из ожидаемого потока, оценка достаточности железа, при необходимости — корректировка заявки DevOps на железо/квоты.
- **RnD часть:**
  - Какая модель / квантизация попадает в SLA на нашем железе.

Условная разбивка по задачам:
1. Замеры throughput/latency модели + расчёт пикового RPS из ожидаемого потока + корректировка заявки DevOps — **1 sp**

**Итого: ~1 sp**

**Зависимости от других команд:**
- DevOps — тестовое железо для замеров; при нехватке ресурсов — заказ дополнительных мощностей.

**Внутренние блокеры:**
- блок 4, п.2 (выбор модели NLU — без него мерить нечего).

### 7. Трекинг через Langfuse

- **Когда нужен / зачем:** наблюдаемость диалога на уровне трейсов — видеть, где пользователь отвалился, какие интенты не распознались, latency LLM-вызовов, связка с ID зацепки в UGC для сшивания с резолюцией модерации. Атрибуты: точка входа, финальный интент, multi-intent, причина fallback, тип зацепки, было ли фото, авторизован ли.

2. Разметка трейсов и атрибутов + маскирование PII + связка с UGC submission ID — **1 sp**

### 8. Мониторинг и алертинг

- **Когда нужен / зачем:** `/metrics` для Prometheus + алерты. Технические: latency p95 под SLA 1–2 сек, error rate, UGC 5xx, LLM-токены и стоимость. Продуктовые: доля fallback на классическую (KPI ≤30%), доля «Другое» / «не распознано», конверсия по этапам воронки. Дашборды Grafana тоже наш скоуп (по операционным метрикам).
- **Суть работы:** эндпоинт `/metrics`, технические + продуктовые + cost-метрики, правила Alertmanager.

Условная разбивка по задачам:
1. `/metrics` + технические метрики (latency p50/p95/p99, error rate, RPS, UGC 5xx) + продуктовые (fallback rate, intent_not_recognized, dialog_abandoned, signal_submitted) + LLM cost — **1 sp**
2. Правила Alertmanager в канал команды (p95 > 3 сек 5 мин, error rate > 5% 5 мин, UGC 5xx > 10/мин, fallback rate > 40% за час, `up==0`) — **1 sp**

**Итого: ~2 sp**

**Зависимости от других команд:**

**Внутренние блокеры:**

## Что вне скопа (явно)

- OCR с фото и распознавание голоса — версионность ценности из брифа, не MVP.
- Всё, что требует пина на карте: входы в здания и фирмы, расположение и форма объектов на карте — откат на классическую форму без переноса полей.
- Карточка остановки/станции метро, «какое-то местечко».
- Длинная память сессии («закрыл — значит закрыл»).
- Нагрузочные тесты — поток пользователей маленький (≈400k нажатий «Сообщить об ошибке» в месяц суммарно по РФ).
- Сборка дашбордов Grafana — на команде observability.
- Маркетинг и редизайн иконки «Сообщить об ошибке» (предложение Ильи Макухи в комментариях брифа) — за пределами нашего скопа.
