# Частые компоненты декомпозиции

Подборка типовых блоков, которые встречаются в большинстве проектов AI-агентов команды. Не все обязательны — но если в проекте есть похожее, **начинай с этих оценок** и подстраивай под специфику.

Калибровка по всему файлу: **1 sp = 1 человеко-день** middle/senior разработчика (~3–6 ч активной работы).

---

## 1. Архитектурная проработка / ТЗ

**Формулировка для названия:** «Формализация потока данных между всеми компонентами, формализация пользовательских сценариев для последующей spec-driven разработки.»

**Что входит:**
- описание состояний / фаз агента, slot-схем по типам задач, депенденси между слотами;
- формализация пользовательских сценариев — happy-path, ветки, fallback-правила, multi-intent логика;
- правила маршрутизации, приоритеты;
- согласование с продактом по неочевидным веткам.

**Типовая оценка: ~3 sp.** Размывается на ~неделю по календарю из-за итераций согласования.

**Примеры калибровки оценки:**

- **AIA-1355 — мониторинг Otello (узкий бриф, один смежник):** один продакт Даша, понятный домен, бриф из 4 пунктов. ТЗ выходит в **одну рабочую неделю / 3 sp** с разбивкой «1 sp спека + 1 sp игрушечный отчёт на 1–2 прошлых неделях + 1 sp итерация согласования за 1–2 звонка».
- **Signals-agent — отправка зацепок (широкий бриф, ≥3 смежника):** фронт + UGC + продакт. ТЗ — 3 sp, но согласования и контракты со смежниками разносятся отдельным блоком на **2 разные недели** (см. блок 2 ниже), потому что звонки и итерации не помещаются в одну.
- **Маркет-агент — каталог товаров (широкий бриф, ML-компонент):** продакт + аналитика + Маркет + поставщик данных. ТЗ растёт до 4–5 sp с разносом на 2 недели по той же причине.

**Нижняя граница — 3 sp.** Даже самое маленькое ТЗ редко делается быстрее: написать → дать прочитать → созвон → правки → принять. Меньше 3 sp ставить — оптимистичная иллюзия, она расползается на календаре в реальности.

**Почему отдельный блок:** если арх. проработка сделана отдельно — реализационный блок («диалоговое ядро») становится **только кодом по уже формализованной схеме**, и его оценка падает в 2-3 раза. Без неё код вынужден вырастать вместе со схемой → больше переделок → больше sp.

---

## 2. Согласования и контракты со смежниками

**Что входит:**
- свод требований к API смежников + созвоны (UGC, фронт и т.д.);
- фиксация контрактов в Confluence;
- доступы в sandbox, тестовые ID.

**Типовая оценка: ~2 sp.** Это **две задачи по 1 sp** с одинаковым содержанием, обязательно разнесённые на **2 разных недели** — потому что звонки и итерации согласования физически не помещаются в одну неделю.

1. Согласования и контракты со смежниками: звонки, фиксация в Confluence, доступы — **1 sp** *(неделя 1)*
2. Согласования и контракты со смежниками: продолжение — **1 sp** *(неделя 2, содержание то же самое)*

Если агент сильно сложнее и контрактов много — оценка может вырасти до 3-4 sp, но всё равно бьётся на 1 sp в каждой неделе.

**Что НЕ входит** (это напоминалки, **без sp**):
- «уточнить у фронта тех. возможность Х»;
- «написать заявку DevOps на сервера».

Эти пункты фиксируй в декомпозиции с пометкой «напоминалка, без sp» — координация важна, но это не часы работы.

---

## 3. Контекст точек входа и маршрутизация (саджесты)

**Что входит:**
- парсер контекста от фронта (точка входа, ID/тип объекта, координаты);
- словарь саджестов по точкам входа;
- маршрутизация саджестов на типы задач;
- флаг «уйти в смежную систему / классическую форму».

**Типовая оценка: ~2 sp.** Большинство правил приходят из таблиц брифа и архитектурной проработки — задача в основном про код маршрутизации, не про решение «что куда».

---

## 4. Диалоговое ядро (реализация агента)

**Что входит:** state-machine, slot-filling, NLU, парсеры свободной формы, fallback-логика, multi-intent, тексты ответов агента (в рамках работы над промптами, отдельным блоком не вынимаются), сборка финального payload для смежной системы.

**Типовая оценка с отдельной арх. проработкой: ~8-10 sp.** Если арх. проработки нет — выйдет в 2-3 раза больше, потому что схемы и правила нужно вырабатывать в процессе кодинга.

**Внутренняя разбивка примерно такая:**
- скелет state-machine + session store — 2 sp
- реализация slot-schemas (формализованных в арх. проработке) — 1 sp
- NLU: классификатор + extractor — 2 sp
- парсеры свободной формы (расписание, телефоны, URL и т.п.) — 2 sp
- сверка с данными источника + подтверждения — 1 sp
- multi-intent, fallback, смена намерения, сборка payload — 2 sp

Тексты ответов агента в **отдельный блок не выделять** — это часть промптинга в рамках этого же ядра.

---

## 4a. Свободное LLM-суждение (классификация, суммаризация, ранжирование)

**Когда нужен:** в задаче появляется LLM-кусок без жёстко детерминированного выхода — модель сама решает в неформализованную таксономию, пишет суммаризацию, ранжирует, выносит вердикт по свободному критерию. Это **не** про вызов API с structured output по фиксированной схеме — это про куски, где результат нужно «сверить с реальностью».

**Что входит:**
- первый чёрновой проход (промпт + structured output / прямой ответ);
- сверка результата на ≥10–20 реальных кейсах с продактом или другим экспертом;
- минимум одна итерация правок промпта/таксономии после сверки;
- закрепление эталонных кейсов как мини-датасет для регрессии при будущих изменениях промпта.

**Типовая оценка:** «детерминированная часть» + **дополнительно ~2 sp** на итерации калибровки. То есть если бы это была функция с заданной формулой — 1 sp; со свободным LLM-суждением — 3 sp.

**Примеры:**

- **AIA-1355 — классификация трейсов в таксономию Даши (false_promise / неполный ответ / нарушение логики / баг / is_new):** 1 sp на первый чёрновой LLM-проход + 2 sp на калибровку с Дашей на реальных трейсах = **3 sp** в составе блока «агент-аналитик». Без калибровки модель промахивается на новых для неё категориях (false_promise, нарушение логики), которые не совпадают 1-в-1 с существующими скорами судьи.
- **AIA-606 — LLM-судья качества (user_satisfaction / agent_perf / hallucinations / user_request_is_new):** на момент пилота промпт был «правильным» формально, но судья штрафовал агента за корректное поведение (отсутствующий тул, цитирование `context_info` как критерия) — пришлось добавить секции «Agent directives» и «injected context is NOT an evaluation criterion» после прогона на 19 трейсах. Это и есть та самая итерация калибровки, без которой LLM-кусок не работает.
- **HotelSummaryGenerator (Otello search):** промпт с жёстким приоритетом фактов — особенности отеля → локация (сильные факты) → номера → окружение. Калибровка шла на сотнях саммари: иначе модель упоминала цену/рейтинг/завтрак (запрещено) или генерировала размытую локацию («парк рядом» вместо «Парк Горького»).

**Что НЕ считается «свободным LLM-суждением»** (этой надбавки не получает):
- вызов LLM с жёсткой Pydantic-схемой по понятным enum-полям, где правила однозначные (как `BaseEval.user_satisfaction` после уже сделанной калибровки — повторный вызов в продакшене);
- prompt-template с подстановкой переменных без свободы формулировок;
- классификация в небольшой закрытый список с очевидными примерами.

## 5. Интеграция со смежной системой (API)

**Что входит:** HTTP-клиент, сериализация по типам задач, валидация payload, ретраи/обработка ошибок, интеграционные тесты против sandbox.

**Типовая оценка: ~4 sp** для среднего API с 5-10 типами запросов. Дробить:
- клиент + сериализация — 2 sp
- валидация + ретраи + загрузка файлов (если есть) — 1 sp
- интеграционные тесты против sandbox — 1 sp

---

## 6. Расчёт нагрузки и подбор железа под локальную модель

**Когда нужен:** новых агентов сразу оцениваем с учётом запуска на локальных моделях. До раскатки нужно понимать, какую нагрузку модель держит на доступном железе, помещаемся ли в latency-SLA при пиковом потоке, нужно ли заказывать дополнительные ресурсы (GPU / CPU / RAM).

**Что входит:**
- замеры throughput и latency выбранной модели на тестовом железе;
- расчёт пикового RPS из ожидаемого потока пользователей;
- оценка достаточности текущего железа;
- при необходимости — корректировка заявки DevOps на доп. ресурсы.

**Типовая оценка: ~1 sp.** Замеры делаются после того, как модель уже выбрана (в рамках NLU/диалогового ядра), поэтому это короткий технический расчёт, а не отдельное RnD.

**Зависимости:** DevOps — тестовое железо для замеров и потенциальный заказ дополнительных мощностей.

---

## 7. Трекинг через Langfuse (или аналог)

**Что входит:** подключение SDK, спан-иерархия трейсов, атрибуты, маскирование PII, связка с ID финального действия в смежной системе.

**Типовая оценка: ~2 sp.**
- день — поднять и настроить SDK, изолировать окружения;
- день — прописать спан-иерархию и атрибуты для трейсов.

---

## 8. Мониторинг и алертинг

**Что входит:** `/metrics` для Prometheus, технические метрики (latency, error rate, LLM-стоимость), продуктовые метрики (fallback rate, конверсия), правила Alertmanager.

**Типовая оценка: ~3 sp.**
- технические + продуктовые метрики — 2 sp
- алерты — 1 sp

Дашборды Grafana, как правило, **не наш скоп** — их собирает команда observability. Зафиксировать это явно в блоке.

---

## 9. Eval-сценарии (регрессия диалога)

**Что входит:** реализация паттерна slot-filling eval (`Checkpoint` + `ScenarioStep` + `Scenario` + `verify_checkpoint` + runner) → см. [knowledge-page «Архитектура eval slot-filling агентов»](../../../../knowledge-pages/eval-slot-filling.md), стабы, сценарии под MVP-кейсы, подключение в CI.

**Типовая оценка: ~3-4 sp**, если паттерн в команде уже описан и переносится из готового. С нуля будет +2-3 sp на каркас.

- каркас + стабы — 2 sp
- сценарии под MVP — 1 sp
- CI-подключение — 1 sp

---

## 10. Операционное тестирование на реальных данных

**Что входит:** прогон end-to-end на легаси-данных (выгрузка старых запросов), фиксация результатов в коротком отчёте. Без подробностей — просто «убедиться, что в целом работает».

**Типовая оценка: ~2 sp.**
- получить выгрузку и привести в формат — напоминалка / 0–1 sp
- прогон + отчёт — 2 sp

---

## 11. Деплой, фича-флаг, пилот

**Что входит:** CI/CD пайплайн, секреты, окружения, антиабуз (для агентов с анонимными сессиями), baseline KPI, план пилота, постпилотный ретро.

**Типовая оценка: ~2 sp.**
- CI/CD + секреты + окружения + антиабуз — 1 sp
- baseline + план + ретро — 1 sp

Фича-флаг на стороне фронта = **чужая работа**, не наш sp. Раскатка 1% → 10% → 50% → 100% — это процесс с продактом, а не задача разработчика.

---

## 12. Буфер на пост-релизные правки

**Что входит:** после раскатки в прод часть проблем неизбежно вылезает — баги от QA, реальное поведение пользователей, edge-кейсы, мелкие правки промптов по сигналам из мониторинга и трейсинга. Без заложенного буфера эти правки съедают следующий спринт.

**Типовая оценка: ~2 sp.**

У этого блока, как правило, **нет зависимостей от других команд** — это резерв, и его делает та же команда, которая выкатывала агента. Один из немногих блоков, где честное «зависимостей нет» — норма, а не сигнал недодуманного сценария.
