# brief-decomposition

Бриф — это запрос на фичу от продакта. Декомпозиция — превращение брифа в список инженерных блоков и подзадач с оценками в sp, блокерами и зависимостями. На этом этапе у нас уже есть хорошее понимание брифа и дополнительные материалы. К брифу заданы вопросы и собраны ответы. Пора фиксировать договорённости и идти в реализацию. Важно понимать, что какие-то неопределённости могут быть не закрыты — это нормально. Наша задачка подсветить их в «открытых вопросах», «блокерах», «рисках».

Декомпозиция строится **от сценария**: бриф описывает фичу или техническое изменение, но и то, и другое существует, чтобы появиться в каком-то сценарии — последовательности шагов и развилок, через которую проходит актор (пользователь / запрос / событие / тренировочный цикл). Каждый инженерный блок возникает потому, что какой-то шаг или ветка сценария требует его реализации.

## Процесс

### Фаза 0. Собрать контекст (молча)

1. Прочитать бриф и все приложенные документы.
2. Через Explore-субагента пройтись по кодовой базе, если она есть: найти существующие сервисы / модули / файлы / сценарии, которых касается бриф. Цель — собрать конкретные точки касания, не пересказ архитектуры.
3. Проверить `claude-tasks/` в проекте на релевантные сводки прошлых задач.
4. Если в материалах брифа (Q&A, доп. документы, заметки) уже есть реконструированный или явный сценарий — переиспользовать как основу для Фазы 1, а не выдумывать свой.
5. Короткое резюме пользователю (5-10 строк, не больше): что затрагивает бриф в коде, какие очевидные новые компоненты, какие части источников друг друга дополняют. Без рассуждений.

### Фаза 1. Восстановить сценарий и связать с компонентами

Восстановить сценарий — последовательность шагов и развилок, через которую проходит **актор**. Актор — тот, чьим путём мы идём, и он не обязательно «пользователь приложения». В зависимости от типа брифа это может быть:

- **конечный пользователь приложения** — самый частый случай (вошёл → увидел саджесты → ввёл текст → подтвердил → отправил);
- **внутренний пользователь** — дежурный инженер, оператор, аналитик (получил алерт → открыл дашборд → проверил логи → принял решение);
- **запрос или событие в пайплайне** — для брифов про инфраструктуру/миграции (запрос пришёл в шлюз → адаптер → модель → постпроцессинг → ответ);
- **тренировочный или деплойный цикл** — для ML/инфра-брифов (данные собрались → отфильтровались → попали в тренировку → метрики → выкатка → откат при деградации).

Внутри сценария различаем **шаги** (последовательные стадии пути актора), **ветки** (развилки внутри шага: что если данные X / Y; что если предыдущий шаг вернул успех / ошибку / пусто) и **граничные состояния** (актор прервал поток и вернулся, смежная система недоступна, контекст пустой, данные не соответствуют формату).

Источники в порядке приоритета: явный сценарий в брифе → сценарий, уже реконструированный в Q&A / сопроводительных документах → собственная реконструкция из списка фич/техизменений (в этом случае в артефакте явно пометить: «сценарий — наша реконструкция, подтверди или поправь»).

Структура — три секции, адаптивно (не все нужны для каждого брифа):

- **A. Главный поток (A1, A2, ...)** — happy-path актора, шаги по порядку.
- **B. Ветки и исключения (B1, B2, ...)** — развилки, fallback'и, edge-cases, граничные состояния данных, недоступность смежной системы. Каждая ветка привязана к шагу A, в котором возникает.
- **C. Технические потоки (C1, C2, ...)** — потоки не от пользователя: трекинг сессий, мониторинг, регрессия в CI, релиз/пилот. Опционально — только если у брифа есть техническая инфраструктура, обслуживающая основной поток.

Для каждого шага / ветки выписать, какие компоненты системы в нём задействованы. Компоненты — связные единицы работы (новый сервис / переработка существующего модуля / новый сценарий / интеграция со смежной командой). Список компонентов проступает из самого сценария: ровно то, что нужно реализовать, чтобы поток заработал.

Формат:

```
## Сценарий и затрагиваемые компоненты

### A. Главный поток

Актор — <конечный пользователь / запрос в пайплайне / ...>.

| # | Шаг / событие | Компоненты |
|---|---|---|
| A1 | <короткое описание шага> | №1 <название>, №3 <название> |
| A2 | ... | ... |

### B. Ветки и исключения

| # | Ветка / событие | Возникает в | Компоненты |
|---|---|---|---|
| B1 | ... | A4–A6 | №2 ..., №4 ... |

### C. Технические потоки

| # | Поток | Компоненты |
|---|---|---|
| C1 | ... | №6 ... |
```

Адаптивно по размеру:
- узкий бриф (один актор, мало компонентов) → A из 3-5 шагов, B и C пустые или с одной-двумя строками;
- широкий бриф → несколько секций A под разных акторов, развёрнутые B и C.

### Фаза 2. Описать блоки

По таблице — каждый компонент превращается в блок. Один блок = один компонент из сводной таблицы.

```
### N. <Заголовок блока — конкретный>

- **Шаги:** A1 (короткое уточнение), A7, A8; ветки B3.
- **Когда нужен / зачем:** 2-3 предложения, привязка к шагам сценария и/или коду.
- **Ответственная команда:** МЫ / другая команда / TBD.
- **Открытые вопросы:** список, если есть. Если нет — секцию опустить.
```

Поле **Шаги** — обязательное; это обратная ссылка из блока в сценарий. Для блоков, обслуживающих только технический поток, заменить на `**Технический поток:** C1`.

Никаких подзадач, никаких оценок, никаких блокеров на этой фазе. Это черновик для согласования границ.

Сохранить файл (см. «Артефакт»), напечатать путь, остановиться и ждать. Не продолжать без явного «ок» / «правь так-то».

### Чекпоинт

Пользователь правит файл руками или говорит, что добавить / убрать / переразбить — как в сценарии (шаги/ветки), так и в блоках. Скилл применяет правки (включая пересчёт связей в сводной таблице, если меняются блоки или шаги) и снова ждёт «ок» — или сразу переходит дальше, если пользователь явно сказал «дальше».

### Фаза 3. Подзадачи

Для каждого утверждённого блока расписать:

```
### N. <Заголовок блока>

- **Шаги:** A1, A7, A8; ветки B3.
- **Когда нужен / зачем:** ...
- **Суть работы:** 2-4 предложения на входе/выходе.
- **Ответственная команда:** ...
- **RnD часть:** (списком, если есть)
- **Открытые вопросы:** (если есть)

RnD-часть – это описание тех вещей, которые нужно будет решить уже в процессе работы над блоком. Например, будем ли мы использовать для маршрутизации llm, локальную или проприетарную? Как именно мы будем следить за зоопарком моделей? Вообще на каждый RnD вопрос скорее всего предполагается отдельная задача на ресёрч.

Условная разбивка по задачам:
1. <конкретная подзадача> — N sp
2. ...

**Итого: ~M sp**

**Зависимости от других команд:** (фронт / UGC / DevOps / продакт / аналитика; если нет — пиши «нет»)
- ...

**Внутренние блокеры:** (от других наших блоков — опционально, если есть)
- блок N — ...
```

## Пример компонента из финальной разбивки

### 2. Поддержка новых рубрик в классификаторе товаров

- **Шаги:** A2 (иерархический проход — рубрика → подкатегория); C1 (обучение/дообучение по новым рубрикам), C2 (мониторинг качества в проде).
- **Когда нужен / зачем:** когда продакт/аналитики приносят новые рубрики (фитнес, спорт, стройка) и деревья к ним. Без расширения классификатора эти рубрики не подключить.
- **Суть работы:** на вход — товар (карточка/описание/фото). На выход — категория. Возможно всего нужен иерархический подход: сначала рубрика, потом классификатор внутри неё (плоский на ~800 категорий работает плохо).
- **Ответственная команда:** AI-агенты.
- **RnD часть:**
  - Иерархический vs плоский классификатор — финальное решение.
  - Версионирование классификаторов, валидация качества, способ выкатывания обновлений.
  - Как мерить качество в продакшене.
- **Открытые вопросы:**
  - Какой RPS наш классификатор поддерживает сейчас и какая у него скорость? Это нужно, чтоб рассчитать, как быстро мы сможем индексировать кучу товаров из маркета + насколько нам надо масштабироваться, чтоб успеть в определённый срок.
  - Готовых размеченных данных по новым рубрикам в Маркете может не быть, а значит, придётся размечать как-то иначе — оцениваем как риск, точечно решаем после выбора категорий.

Условная разбивка по задачам:
1. сбор и подготовка обучающих данных по новым рубрикам (наложение наших деревьев на категории Маркета, как сейчас делает Антон) – 3 sp
2. выбор подхода, тестирование 1 (классические классификаторы / берты / иерархия?, как сейчас) – 4 sp +
3. выбор подхода, тестирование 2 (LLM проприетарные и локальные, маленькие и большие, расчёты стоимости / latency / RPS) – 4 sp
4. обучение/дообучение под новые рубрики, замеры качества – 4 sp
5. версионирование / зоопарк моделей – 2-3 sp
6. мониторинг качества в проде (метрики, алерты) – 2 sp

**Зависимости от других команд:**
- продакт + аналитика — финальный список новых рубрик и их деревьев;
- Маркет — доступ к данным для подготовки обучающих наборов (как сейчас делает Антон); для новых рубрик данных может не быть.

**Внутренние блокеры:**
- результат экспериментов из §1 — если выберется LLM-подход, классификатор может пойти туда же.

## Принципы

**1. Конкретика, не общие слова.** Каждый блок и подзадача упоминают конкретные сущности из брифа / созвонов / кода / заметок и явно ссылаются на шаги сценария. Если формулировка общая и её можно без изменений вставить в любой другой бриф — переписать. «Реализовать сервис обогащения» — плохо. «Обогащение карточек товара атрибутами по дереву слотов, выход — JSON в индекс прокси» — хорошо.

**2. Блок без шага — флаг.** Если блоку нечего написать в `**Шаги:**`, скорее всего это лишний блок или сценарий описан не полностью. Не молчать — поднять с пользователем на чекпоинте.

**3. Фаза 2 — лаконично.** До чекпоинта никаких длинных описаний, RnD-частей, разбивок на подзадачи. Только заголовок + шаги + 1-2 строки сути + ответственная команда + (если есть) открытые вопросы. Соблазн расписать побольше — глушить.

**4. Истина = бриф + Q&A + треды + код. Этого достаточно.** Полной определённости не будет и не нужна — двух недель на выяснение каждой детали у нас нет. Работаем с тем, что есть. Дырки — фиксируем в `Открытые вопросы`, не пытаясь добиться полноты.

**5. ≤4 sp на подзадачу.** Больше — дробить. Меньше нижней границы (например, 1 sp) — обычно объединять с соседней или прятать в «прочее», если это не критическая отдельная штука.

**6. Блокеры — это в первую очередь про другие команды.** Самая полезная для планирования информация в секции блокеров — зависимости от смежных команд (фронт, UGC, DevOps, продакт, аналитика). Их выделяй явно — для каждого блока понятно, кто его блокирует со стороны. Внутренние зависимости от других наших блоков — отдельной подсекцией ниже, если есть. Если внешних зависимостей нет — пиши «нет» прямым текстом: это полезный сигнал, что блок можно делать параллельно с согласованиями.

## Оценки

Шкала — story points, как в команде.

**Базовая калибровка: 1 sp = 1 человеко-день middle/senior разработчика (~3–6 часов активной работы).** Это про усилия, а не про календарь — задача на 1 sp легко растягивается на 2–3 календарных дня из-за созвонов, ожиданий, контекст-свитчей.

Ориентиры:
- 1 sp — тривиальная интеграция, маленький кусок кода с очевидной реализацией.
- 2-3 sp — небольшие эксперименты, значительные изменения в коде, заметная связная задача.
- 4 sp — крупный кусок: подход с экспериментами, новый сервис.
- >4 sp — дробить.

RnD-задачи («выбор подхода, тестирование») почти всегда 3-4 sp — это не часы кода, а серия проб; занижать ниже 3 sp не стоит, но и завышать «потому что страшно» — тоже.

**Калибровка по типу проекта.** Если проект относится к классу, для которого есть пример в `examples/`, **начинай с указанных там оценок** и подстраивай под специфику. Не оценивай частый компонент с нуля — это путь к разбросу в 2-3 раза.

## Что не получает оценки

Не каждый пункт декомпозиции — задача разработчика. Различаем три типа единиц:

1. **Задачи** — то, что разработчик делает руками. Получают sp.
2. **Напоминалки** — «уточнить у X», «написать заявку Y», «дождаться ответа Z». **Не получают sp.** Это не часы работы, а пункт в чеклисте, чтобы не забыть. Фиксируй их в декомпозиции — координация важна — но **без оценки**. Помечай словом «напоминалка» или «без sp».
3. **Чужая работа** — то, что делают смежники (фронт, UGC, DevOps). В sp нашей команды не входит, фиксируется как блокер/зависимость.

Типовые ошибки: «уточнить у фронта про возможность кнопки» = 1 sp; «отправить заявку DevOps» = 1 sp. Это не задачи, это пара сообщений в Slack.

## Частые компоненты

У большинства проектов AI-агентов в декомпозиции повторяется один и тот же набор блоков с похожими оценками. Не изобретай оценки с нуля — смотри в [examples/common-components.md](examples/common-components.md). Там перечислены архитектурная проработка, согласования, контекст и маршрутизация, диалоговое ядро, интеграция со смежной системой, трекинг, мониторинг, eval, операционное тестирование, деплой и пилот.

См. также [examples/](examples/) — реальные декомпозиции после калибровки.

## Артефакт

Один markdown-файл. По умолчанию — рядом с брифом, имя `decomposition_<краткое-имя-проекта>.md` (имя проекта вывести из контекста). Можно явно передать путь третьим аргументом, тогда сохранить туда.

Структура файла:
1. Заголовок + источники + краткая преамбула.
2. Секция «Сценарий и затрагиваемые компоненты» — таблицы A / B / C.
3. Секция «Компоненты» — блоки по номерам из таблицы, каждый со строкой `**Шаги:**`.
4. Секция «Что вне скопа (явно)» — короткий список того, что не делаем, со ссылками на ветки/потоки (например, «B3 — откат на классическую форму»).

Файл строится итеративно: фазы 1+2 пишут каркас (источники + сценарий + черновик блоков с шагами), фаза 3 дописывает «Суть работы», RnD, подзадачи и блокеры в те же секции, не пересоздавая файл.
